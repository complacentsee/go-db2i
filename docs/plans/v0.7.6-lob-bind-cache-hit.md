# v0.7.6 — LOB-bind cache-hit fast path + v0.7.5 cleanup

## Context

v0.7.5 (tagged 2026-05-11, commit `d7d42d1`) closed the last bug
class around the cache-hit dispatch: SQL-204 / SQL-805 fallback on
DDL invalidation, graceful fallback on the LOB-bind encoder gap,
and PUB400 V7R5M0 cross-LPAR validation. The forward-commit
password scrub + history rewrite (filter-repo) shipped alongside.

`TestLOBBind_FilingProbe` in
`test/conformance/cache_hit_test.go:1789-1923` (the empirical probe
that overturned the v0.7.4 "JT400 filters LOB-bind" inference)
proved three things on IBM Cloud V7R6M0:

1. The server DOES file LOB-bind INSERTs into the `*SQLPKG`.
   `SYSPACKAGESTAT` shows `NUMBER_STATEMENTS=1` after the 4-iter
   loop crosses the 3-PREPARE threshold; `SYSPACKAGESTMTSTAT`
   captures the renamed name (e.g., `QZAF491DC28F222001`).
2. Our v0.7.4 auto-populate path correctly downloads the renamed
   name on iter 4 and attempts cache-hit dispatch.
3. The dispatch hits `ErrUnsupportedCachedParamType` because the
   `*PGM`-stored `ParameterMarkerFormat` carries the **raw-LOB**
   SQL types (404/405/408/409/412/413), while the cache-hit
   encoder (`hostserver.EncodeDBExtendedData` in
   `hostserver/db_prepared.go:335-351`) only has a branch for the
   live-PREPARE **locator** SQL types (960/961/964/965/968/969).

v0.7.5 added the graceful fallback (purge + re-PREPARE) so the
LOB-bind path keeps working end-to-end. v0.7.6 closes the
remaining gap: making cache-hit dispatch actually fire for LOB-bind
statements, saving one PREPARE_DESCRIBE round-trip per call.

Phase-1 research summary (saves a fresh session re-discovery):

- The raw-LOB SQL types 404/405/408/409/412/413 are
  **semantically equivalent to the locator types 960/961/etc for
  cache-hit dispatch**. Both ultimately encode as a 4-byte locator
  handle in CP `0x381F`. The only difference: the *PGM-stored
  shape carries the column's declared max size in `FieldLength`
  (e.g. 65536 for `BLOB(64K)`), while the live-PREPARE reply
  always returns `FieldLength=4` for locator types.

- `ExecutePreparedCached` (`hostserver/db_cached.go:49-158`) has
  no WRITE_LOB_DATA logic today — it assumes parameter values are
  inline-encodable. The regular-path equivalent (`bindLOBParameters`
  at `hostserver/db_lob.go:306-357`) is reusable.

- `prepared_blob_insert.trace` and `prepared_blob_batch.trace`
  cover JT400's regular-path (non-cached) BLOB INSERT wire shape;
  **no existing fixture captures cache-hit dispatch for a LOB-bind
  statement** — hence Part A captures one.

Tight scope (user-confirmed): LOB-bind + cleanup. Niches (DECLARE
PROCEDURE filing, `package-criteria=extended`) stay deferred.

## Approach

### Part A: JT400 reference fixture `PackageFilingLOBCacheHit`

Empirical-first (user-confirmed). Capture JT400's exact wire shape
for a 4-iter LOB-bind INSERT under extended-dynamic before writing
the Go code. Same pattern as the v0.7.4 `PackageFilingIUD` fixture.

New case in
`testdata/jtopen-fixtures/src/main/java/io/github/complacentsee/db2i/fixtures/Cases.java`,
mirroring `PackageFilingIUD` (line 1257) but with a BLOB column:

```java
private static final class PackageFilingLOBCacheHit extends WithPackage {
    private static final String TABLE_SHORT = "GOJTFLOB";
    private static final int PREPARE_COUNT = 4;
    PackageFilingLOBCacheHit(String schema) {
        super("prepared_package_filing_lob_cache_hit", schema);
    }
    @Override public void setup(Connection conn) throws SQLException {
        // DLTOBJ + CREATE TABLE schema.GOJTFLOB (ID INT, PAYLOAD BLOB(64K))
    }
    @Override public void execute(Connection conn, GoldenWriter golden) throws SQLException {
        // INSERT INTO schema.GOJTFLOB (ID, PAYLOAD) VALUES (?, ?)
        // 4 iters, each with a 1 KB deterministic BLOB
        // Then SELECT FROM SYSPACKAGESTAT + SYSPACKAGESTMTSTAT
    }
}
```

Register at `Cases.java:124` next to `cases.add(new PackageFilingIUD(schema));`.

Capture via `mvn -q exec:java` against IBM Cloud V7R6M0 (existing
jump-host workflow; `.env.ibmcloud-v7r6` sources cleanly). The
trace will answer:

- **Does JT400 emit `WRITE_LOB_DATA` before each cache-hit
  EXECUTE?** If yes, our implementation needs to upload bytes
  every call. If no (somehow), JT400 found a way to ship inline.
- **Does the cache-hit EXECUTE carry CP `0x3806` (statement-name
  override), or does it use CP `0x3804` (package marker) like
  the v0.7.4 IUD path?** Both work server-side; matters for
  byte-equivalence assertions.
- **What CCSID + length fields appear in the EXECUTE's CP `0x381F`
  payload for the LOB parameter slot?** Confirms the 4-byte
  locator-handle shape vs anything weirder.

Decode via the existing `hostserver.DecodeDBRequestFrame` utility
(introduced in v0.7.4 at `hostserver/db_request_decode.go`).

Deliverable: `prepared_package_filing_lob_cache_hit.trace` +
`.golden.json` committed alongside the existing fixtures, plus a
one-paragraph summary in this plan file documenting JT400's wire
shape for cache-hit LOB-bind.

### Part B: Implementation — encoder branch + WRITE_LOB_DATA dispatch

Two surgical additions:

**B.1: Encoder branch for raw-LOB types.** Extend
`EncodeDBExtendedData` (`hostserver/db_prepared.go:130-379`,
specifically the type switch at lines 215-376) with a new case
matching the locator-type case at lines 335-351:

```go
case 404, 405, 408, 409, 412, 413:
    // *PGM-stored raw-LOB SQL types. Semantically equivalent to
    // the locator types for cache-hit dispatch: the value slot
    // carries the 4-byte server-allocated locator handle, with
    // the actual content already shipped via WRITE_LOB_DATA by
    // ExecutePreparedCached. FieldLength in the cached SQLDA
    // carries the column's declared LOB max (e.g., 65536); the
    // encoder validates and writes 4 bytes regardless -- the
    // EXTENDED-DATA-FORMAT (CP 0x381E) was already updated by
    // CHANGE_DESCRIPTOR to FieldLength=4 to match.
    h, err := toUint32Handle(v)
    if err != nil {
        return nil, fmt.Errorf("hostserver: param %d: %w", i, err)
    }
    be.PutUint32(buf[dataOff:dataOff+4], h)
    dataOff += 4
```

**B.2: WRITE_LOB_DATA dispatch in cache-hit path.** Extend
`ExecutePreparedCached` (`hostserver/db_cached.go:49-158`) to
upload LOB bytes BEFORE the EXECUTE for any slot whose cached
SQL type is a raw-LOB. Insertion point: between the
`sendCachedChangeDescriptor` call (line 73) and the
`EncodeDBExtendedData` call (line 78).

Reuse `bindLOBParameters` (`hostserver/db_lob.go:306-357`) which
already handles the allocate-locator + WRITE_LOB_DATA + return-
handle flow. The helper needs the live-PREPARE shape (locator
types 960/961/etc with FieldLength=4); we'll need to either:

- (a) Translate the cached shape to a synthetic live-PREPARE
  shape at the boundary (`preparedParamsFromCached` extension,
  `hostserver/db_cached.go:343`) — clean, but adds a code path
  the regular flow never touches.

- (b) Build a small `bindLOBParametersFromCached` adapter that
  takes the cached `ParameterMarkerFormat` directly — more code,
  but isolates the cache-hit-specific normalisation
  (FieldLength→4, SQLType remapping if needed).

Pick (a) if Part A's trace shows JT400 hits the same locator
upload path either way; pick (b) if there's a wire divergence
that warrants separate plumbing. Default to (a) on simplicity.

Plus a `CHANGE_DESCRIPTOR` adjustment: when the cached SQLDA
declares a LOB column's max size in `FieldLength`, we need to
override that to 4 (the locator-handle width) in the descriptor
sent to the server, so the server's parser doesn't expect 64K of
inline bytes. This is the analog of how the live `PREPARE_DESCRIBE`
reply always reports `FieldLength=4` for locator types.

**B.3: Recovery on partial WRITE_LOB_DATA failure.** Mirror the
RPB-cleanup pattern already added in v0.7.5 (`db_cached.go:78-91`):
if WRITE_LOB_DATA fails mid-loop, `deleteRPB` before returning so
the driver-layer fallback to plain PREPARE can re-`CREATE_RPB`.

## Out of scope (track separately)

- **`package-criteria=extended`**. JT400's third criterion that
  adds CALL / VALUES / WITH to the filing matrix. ~½ day. No user
  has asked. Deferred.

- **DECLARE PROCEDURE filing.** JT400 files it; we don't route
  through `ExecutePreparedSQL`. Niche; deferred.

- **Cache-hit support for sql.Out (stored procedures with OUT
  params).** Currently `hasOutDest` skips cache-hit for any
  procedure call with `sql.Out` slots. Wire shape diverges further
  than LOB-bind because OUT slots need decoded back into Go
  pointers, which the cache-hit path doesn't yet plumb.
