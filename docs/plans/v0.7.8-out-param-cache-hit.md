# v0.7.8 plan — OUT/INOUT CALL cache-hit fast path

**Status:** speculative — opened 2026-05-12 as a follow-up to the
v0.7.7 `package-criteria=extended` work. The headline question is
whether the cache-hit fast path can be extended to dispatch
OUT/INOUT stored procedures, eliminating the
`refreshPackageCache` skip in v0.7.7 (and the underlying perf gap
that motivated it). The work is not approved; this file captures
the open question so a future session doesn't have to re-derive
it from scratch.

## Why this matters

Under v0.7.7 `package-criteria=extended`, OUT-param CALL
statements file into the `*PGM` but cannot dispatch through the
cache-hit fast path: `preparedParamsFromCached`
(`hostserver/db_cached.go:372`) rejects any cached PMF whose
`ParamType != 0x00 / 0xF0`. Every OUT/INOUT call therefore pays
the full `PREPARE_DESCRIBE` round-trip, even after the server has
filed the statement and even though the cache-hit path has
infrastructure for OUT data (the regular `TestStoredProcedureOUT`
path uses `0xF1` / `0xF2` direction bytes in `CHANGE_DESCRIPTOR`
and decodes OUT values from EXECUTE responses).

v0.7.7's `hasOutDest` refresh-skip is the right behaviour given
the current cache-hit limitation. If the limitation can be lifted,
the refresh-skip can be removed too.

## What we don't know

1. **Does JT400 take the path?** Every JT400 cache-hit capture
   we've collected is IN-only — under JT400 `default`, CALL
   doesn't file, so no OUT cache-hit can exist in those traces.
   We haven't captured a JT400 client running with
   `package criteria=extended` against a procedure with OUT/INOUT
   parameters.
2. **Does the server honor the stored PMF's direction bytes on
   cache-hit?** The cache-hit path overrides the statement name via
   CP `0x3806`. We don't know whether the server then resolves the
   PMF (including direction bytes) from the `*PGM`-stored shape or
   demands the client's `CHANGE_DESCRIPTOR` carry the direction
   bytes explicitly.
3. **What's the right `ParamType` byte on cache-hit
   `CHANGE_DESCRIPTOR` for an OUT slot?** Current code forces
   `0x00` for all slots; for an OUT slot the candidates are
   `0xF1` (OUT) or `0xF2` (INOUT). JT400's behaviour here is
   undocumented.

## Investigation plan

### Part A — JT400 fixture capture (long-pole)

In `testdata/jtopen-fixtures/src/main/java/.../fixtures/Cases.java`,
add a `PackageFilingCallOutCacheHit` case modelled on the existing
`PackageFilingLOBCacheHit`:
- Pass `package criteria=extended` (or whichever JT400 spelling
  maps to our `extended`) on the connection URL.
- Pre-condition: create a deterministic procedure
  `GOJTPCO` with one IN and one OUT parameter inside the test
  schema. Use `CREATE OR REPLACE PROCEDURE` so the fixture is
  idempotent.
- Iter loop ≥ 4 (cross the 3-PREPARE filing threshold).
- On iter 4 (post-threshold), JT400 *should* dispatch via cache-hit
  — capture its wire bytes.

Inspect the resulting `prepared_package_filing_call_out_cache_hit.trace`:
- Iter 4 `CHANGE_DESCRIPTOR` (CP `0x381E`) payload: what direction
  byte does JT400 send for the OUT slot? `0xF1`, `0x00`, or
  something else?
- Is `PREPARE_DESCRIBE` (op `0x1803`) absent on iter 4 (cache-hit)
  or present (cache miss, JT400 not taking the path)?
- If `PREPARE_DESCRIBE` is present every iter (mirroring the v0.7.6
  LOB-bind situation), the answer is **JT400 doesn't take the
  path either** → STATUS=deferred, document and stop.

### Part B — go-db2i implementation (if Part A is promising)

Pivot on Part A's findings:

**If JT400 sends `0xF1`/`0xF2` direction bytes on cache-hit:**
- Relax `preparedParamsFromCached`: accept `0xF1` and `0xF2`,
  preserve the direction byte in the output `PreparedParam.ParamType`
  (instead of forcing `0x00`).
- Confirm `EncodeDBExtendedDataFormat` / `EncodeDBExtendedData`
  emit the right shape for OUT slots on cache-hit.
- Wire up OUT-value decode from the EXECUTE reply in the cache-hit
  result handler (`ExecutePreparedCached` returns `ExecResult`; the
  reply carries `OutValues` for the regular path — confirm the
  cache-hit path either inherits that or needs its own decode).
- Drop `hasOutDest` from the `shouldRefresh` gate in `Stmt.Exec`
  (v0.7.7's defensive skip becomes unnecessary).

**If JT400 sends `0x00` for all slots and the server still
populates OUT values from the stored PMF:**
- Even simpler — keep `ParamType=0x00` on the wire, but stop
  rejecting OUT-direction *input* PMFs. The client-side guard at
  `preparedParamsFromCached` becomes a no-op.
- Drop `hasOutDest` from `shouldRefresh`.

**If JT400 doesn't cache-hit OUT-CALLs (Part A finds
`PREPARE_DESCRIBE` on every iter):**
- STATUS=deferred. Document in this plan + memory; v0.7.7's
  refresh-skip remains the right behaviour. Investigate later if
  we can find a non-JT400 server-side spec or capture.

### Part C — tests and regression coverage

If Part B succeeds:
- New `TestCacheHit_CriteriaExtended_OutCallDispatches` (positive
  case): file an OUT proc through threshold, fresh conn, assert
  `"db2i: exec cache-hit"` slog line + correct OUT-value scan.
- Update `TestCacheHit_OutParameterFallthrough` semantics — it
  currently asserts NO cache-hit for OUT under `default`; that
  remains correct since default doesn't file CALL. The doc comment
  needs adjustment for the extended-criteria case.
- Update `TestCacheHit_CriteriaExtended_OutCallSkipsRefresh`: if
  refresh-skip is removed, this test is no longer meaningful —
  delete and document the removal in CHANGELOG.

## Risks

- **JT400 might not take the path.** Most likely outcome from Part
  A based on the v0.7.6 LOB-bind analogy. Time-box Part A to half
  a day; if `PREPARE_DESCRIBE` is on every iter, STATUS=deferred
  and don't sink time into Part B.
- **Server-side semantics might differ on V7R5M0 vs V7R6M0.** Run
  Part B against both LPARs before tagging.
- **OUT-value decode on cache-hit might require a different RPB
  setup than the IN-only path.** The current
  `OpenSelectPreparedCached` / `ExecutePreparedCached` paths
  diverge from their regular counterparts in subtle ways
  (e.g., `STMT0001/CRSR0001` slot names). Confirm OUT-value
  resolution still works against those slot names; the regular
  CALL path uses different names.

## Effort

- Part A (fixture capture + analysis): ~half-day. Requires mvn +
  JT400 fixture setup (already established in `testdata/jtopen-
  fixtures/`).
- Part B (implementation): ~1 day if Part A is promising.
- Part C (tests + docs): ~half-day.
- Total best-case: ~2 days. Most likely outcome: STATUS=deferred
  after Part A.
